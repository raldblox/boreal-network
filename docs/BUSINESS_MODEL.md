# Business Model

This file defines how Boreal should make money and stay profitable.

## Primary Revenue Model

The primary model is software-mediated work commerce.

That means Boreal should earn revenue first from:

- take rate on funded request GMV

The first durable economic engine is not software seats alone.
It is transactions flowing through the request, commitment, fulfillment, and payout system.

## Secondary Revenue Layers

After core transaction revenue, Boreal can add:

- premium routing or urgency fees
- managed or curated fulfillment layers
- private supply or enterprise network software
- payout, verification, compliance, or API monetization

These should stack on top of the core commerce engine, not replace it.

## What The Public Market Already Proves

- [Upwork](https://investors.upwork.com/news-releases/news-release-details/upwork-reports-fourth-quarter-and-full-year-2025-financial) reported `29%` adjusted EBITDA margin for full-year 2025.
- [Fiverr](https://www.fiverr.com/news/fiverr-q1-2026-earnings) reported `21.4%` adjusted EBITDA margin in Q1 2026.

Inference:

Software-mediated work marketplaces and work platforms can be profitable when matching, trust, workflow, and payments are systematized.

## Boreal's Profitability Rules

To preserve attractive economics, Boreal should follow these rules:

1. Keep the request lifecycle structured.
Structured demand lowers matching and execution cost.

2. Standardize commitment and fulfillment contracts.
Standardization reduces manual ops and makes software automation possible.

3. Treat manual managed services as a support layer, not the core engine.
If humans must handhold every request, margins collapse.

4. Keep proof, payment, and payout first-class.
That is where trust and monetization compound.

5. Let private supply and enterprise lanes raise ACV later.
Do not force the initial business to depend on enterprise sales cycles.

## What To Avoid

Avoid these economic traps:

- becoming a labor-heavy agency
- depending on custom scoping by internal operators
- monetizing only seats before transaction value is proven
- supporting too many low-value or low-trust work classes too early
- taking on fulfillment liability that software should route instead

## Expansion Model

Expected economic progression:

1. self-serve request GMV
2. higher-value repeat workflows
3. curated or premium coordination layers
4. private supply and enterprise software layers

This progression preserves a platform profile instead of sliding into pure services.

## Canonical Monetization Reading

Boreal should be read as:

- commerce first
- software-mediated execution second
- enterprise workflow expansion later

The business is strongest when the software controls the thread and the thread controls the economics.
