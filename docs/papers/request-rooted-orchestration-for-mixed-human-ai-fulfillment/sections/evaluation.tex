The evaluation goal is not only to test whether the system can produce fluent plans. It is to test whether the system preserves executability, approval safety, and proof obligations across both digital and embodied work. To do that, the request-rooted lead-first planner should be compared against at least three baselines: a chat-only orchestration baseline, a task-first planner that decomposes the request immediately, and a direct-tool router that prefers digitally callable actions whenever possible.

The scenario set should include both digital-heavy and embodied or hybrid requests. Digital-heavy scenarios can test continuity, commitment boundaries, and artifact traceability on requests such as workflow automation or research execution. Embodied scenarios should be designed specifically to expose omission risk. Examples include onsite property inspection, local inventory audit, compliance site visit, hardware installation verification, or document pickup with witnessed handoff. Hybrid scenarios can mix local execution with digital documentation or reconciliation.

Several metrics are useful. \emph{Lead-routing accuracy} measures whether the planner selects a credible lead supply for the request. \emph{Over-decomposition rate} measures whether the planner creates unnecessary microtask trees for low-complexity work. \emph{Forbidden mutation rate} measures whether the system creates fulfillment or execution-grade artifacts before approval or outside allowed lanes. \emph{Embodied-step recall} measures whether the generated plan retains execution-critical physical steps. \emph{False-completion rate} measures how often the planner or workflow marks work as effectively complete without the required embodied execution or proof. \emph{Verification completeness} measures whether the final artifact set contains the required type of evidence for the scenario.

Evaluation should also include reviewer-rated executability. Human reviewers can compare plans and outcomes against the scenario requirements and judge whether the work could actually be completed from the produced plan. This is important because some failures will not look obviously wrong at the text level. A plan may sound reasonable while still missing a key access dependency or field verification step.

The current artifact already provides a starting point through deterministic fixtures and an eval runner for extraction, matching, planning, and mutation safety. Extending that baseline with embodied scenarios would allow the paper to test its central claim directly: whether a request-rooted model with explicit embodied constraints reduces generative plan collapse compared with chat-first or tool-first alternatives.
