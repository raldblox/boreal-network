# Boreal Landing Page Messaging Framework

This file defines how Boreal should message the first wedge on a landing page.

It is optimized for:

- startup and SMB buyers
- urgent external digital work
- fuzzy intake
- mixed human-plus-AI execution
- proof, review, and payout continuity

It is not optimized for protocol audiences first.
It is not optimized for generic AI hype.

## Messaging Goal

Make the right buyer say:

`Yes, this is for the kind of work I actually struggle to get done.`

Not:

`Interesting concept.`

## Messaging Priorities

The page must do these in order:

1. make the pain legible fast
2. state the product shape simply
3. explain who it is for
4. show how Boreal is different from chat, marketplaces, and PM tools
5. show trust primitives
6. ask for one concrete action

## Core Message Stack

### Layer 1. Problem

Work starts everywhere and gets lost everywhere.
A serious request begins in chat, docs, email, calls, or prompts, then fragments across people, tools, and partial outputs.

### Layer 2. Product

Boreal turns one request into one accountable fulfillment thread.

### Layer 3. Buyer value

Boreal helps buyers carry messy, high-importance work from ask to done without losing context across routing, execution, proof, review, or payout.

### Layer 4. Category

Lower on the page, not always in the hero:

Boreal is request-native work commerce for mixed human and AI fulfillment.

## Hero Framework

### Recommended Hero Version A

Headline:
`One thread from request to done.`

Subheadline:
`Boreal helps startups and SMBs turn serious work requests into accountable completion across humans and AI, with proof, review, and payout kept on the same track.`

CTA options:
- `Bring one serious request`
- `Request pilot access`
- `Submit a request`

Support line:
`For work AI cannot responsibly finish alone.`

### Recommended Hero Version B

Headline:
`When chat is not enough and marketplaces are too lossy.`

Subheadline:
`Boreal carries high-importance digital work from messy ask to verified closeout, without dropping context between intake, execution, proof, and payment.`

CTA options:
- `Bring your request`
- `Start a pilot`

### Recommended Hero Version C

Headline:
`Bring the messy ask. Keep the whole job on one track.`

Subheadline:
`Boreal is built for startup and SMB teams buying work that needs more than one AI answer and more accountability than a chat thread.`

### Public Solution Variant

Use only when the page is about funded public requests or reusable solutions.

Headline:
`Free to inspect. Paid to run.`

Subheadline:
`Boreal helps people fund real requests, publish accepted artifacts as public solutions, and charge credits only when someone reruns the solution with inference, workflow execution, or review.`

CTA options:
- `Fund a request`
- `View public solutions`
- `Run with credits`

## Above-The-Fold Rules

Must communicate quickly:

- what Boreal does
- who it is for
- why it is different
- what the user should do next

Must avoid above the fold:

- protocol vocabulary
- deep architecture talk
- marketplace grandiosity
- broad claims about replacing all work software
- jargon like `intent-to-fulfillment` before the pain is understood

## Problem Section

### Section goal

Make the user feel seen.

### Suggested framing

`A work request can start anywhere.
But once it matters, most teams still end up stitching together chat, docs, tools, vendors, and AI outputs by hand.
That is where context gets lost, ownership gets blurry, and polished drafts get mistaken for finished work.`

### Alternative bullets

- Chat keeps the conversation, not the accountability.
- Marketplaces help you find people, not carry the whole job through.
- PM tools track tasks after the work is already scoped.
- AI tools draft quickly, but they do not close the loop by themselves.

## Solution Section

### Section goal

Explain Boreal in plain language.

### Recommended structure

`Boreal turns one serious request into one durable work thread.`

Within that thread:
- the request gets clarified
- the right lead gets routed in
- execution stays attached to the original ask
- artifacts and proof stay visible
- review, revision, and payout stay on the same track
- accepted public solutions stay inspectable while live reruns consume credits only when execution capacity is used

### Plain-language value bullets

- Bring one request instead of managing five disconnected tools.
- Keep ownership, proof, and delivery attached to the same job.
- Let AI help where it should, and keep humans accountable where they still matter.

## Who It Is For Section

### Suggested heading
`Built for serious external work, not casual task dumping.`

### Suggested copy

`Boreal fits startup and SMB teams that need urgent digital work done when the ask is still fuzzy, outside help is needed, and proof matters before closeout.`

### Good-fit bullets

- AI automation implementation
- workflow rescue and cleanup
- MVP or prototype execution
- research that must turn into action
- specialist digital work that needs review and proof

## How It Works Section

Keep this short and concrete.

### Suggested 5-step flow

1. Submit the request
2. Clarify the outcome and constraints
3. Route the right lead or specialist path
4. Execute with artifacts, review, and proof attached
5. Close out with acceptance and payout on the same thread

### Rule

Show product truth.
Avoid pretending there is instant universal liquidity if the real launch motion is curated.

## Differentiation Section

### Against chat
`Chat holds words. Boreal holds the job.`

### Against marketplaces
`Marketplaces help with discovery. Boreal helps carry the work through completion.`

### Against PM tools
`PM tools manage tasks after scope is known. Boreal starts earlier, where the ask is still messy.`

### Against AI-only tools
`AI can generate a lot. Boreal is built for work that still needs human judgment, proof, and closeout.`

## Trust Section

This section matters more than polish.

### Must communicate

- request ownership
- lead accountability
- visible artifacts
- proof expectations
- review and revision path
- payout or settlement clarity
- honest launch boundaries

### Suggested heading
`Built for accountable completion.`

### Suggested bullets

- one durable request thread
- clear owner and next step
- visible artifacts and proof
- review before closeout
- payout tied to completion state

## Social Proof / Early Proof Section

Until you have volume, use:

- pilot framing
- screenshots
- request walkthroughs
- before-and-after examples
- narrow testimonials

### Good proof angles

- reduced tool switching
- better visibility on work status
- better proof of completion
- less chasing and restating
- clearer ownership after intake

## CTA Section

Keep it singular.

Best early CTA:
`Bring one serious request.`

Other valid versions:
- `Request pilot access`
- `Submit a request`
- `Test Boreal on a real workflow`

Avoid too many CTA branches early.
The first job is to get the right buyer into one real request flow.

## Tone Rules

Boreal should sound:

- direct
- calm
- specific
- operational
- serious about work
- anti-fake-completion

Boreal should not sound:

- mystical
- overabstract
- protocol-obsessed
- anti-human
- anti-AI
- too marketplace-generic

## Messaging Guardrails

Do say:
- serious request
- completed outcome
- one thread
- proof and review
- mixed human and AI execution
- work AI cannot finish alone
- public solutions are free to inspect
- running a solution consumes credits when inference or execution is used

Do not lead with:
- decentralized network
- agent marketplace
- swarm economy
- ontology-heavy framing
- protocol-depth language
- grand claims about full automation
- pay to reveal public knowledge

## Page Success Criteria

A strong first page should make the right visitor able to answer these questions in under a minute:

- Is this for the kind of work I need done?
- Is this better than my current patchwork?
- Does this seem trustworthy enough to test on a real request?
- What do I do next?

If the user leaves only thinking `interesting concept`, the page is too abstract.
